# 06. Языки и AI-интеграция: plpython3u + BAML

> Важно: это **два разных слоя**.
> - **plpython3u** — это **расширение PostgreSQL** (untrusted процедурный язык). Даёт возможность писать хранимые функции на Python 3 прямо в БД.
> - **BAML** — это **Python-библиотека** (не расширение PG). Ставится через `pip` и **вызывается из тела** функций plpython3u, чтобы типобезопасно вызывать LLM и получать структурированный вывод.
>
> Связка: SQL-функция `LANGUAGE plpython3u` → внутри тела импортирует сгенерированный `baml_client` / `baml_py` → вызывает LLM → возвращает в SQL типобезопасный результат.

---

## plpython3u

### Назначение
Процедурный язык PostgreSQL, позволяющий писать хранимые функции на Python 3. Язык **untrusted** (суффикс `u`): функции выполняются с полными правами серверного процесса, поэтому их создание разрешено **только суперпользователю**.

### Ключевой SQL API
Установка языка:
```sql
CREATE EXTENSION plpython3u;
```
Создание функции:
```sql
CREATE FUNCTION add(a integer, b integer)
RETURNS integer
LANGUAGE plpython3u
AS $$
    return a + b
$$;
```

В теле функции доступен модуль **`plpy`** и два глобальных словаря:
- `plpy.execute(query_or_plan[, args])` — выполнить SQL, возвращает список dict-подобных строк.
- `plpy.prepare(query, argtypes)` — подготовить план; затем `plpy.execute(plan, args)`.
- `plpy.cursor(query)` — курсор для больших выборок.
- `plpy.subtransaction()` — менеджер контекста для вложенных транзакций (откат при ошибке).
- `plpy.commit()` / `plpy.rollback()` — управление транзакцией (если вызов не в верхнеуровневой транзакции).
- Логирование: `plpy.debug/info/notice/warning/error/fatal/log(msg)`.
- Исключения: классы `plpy.SPIError` (и вложенный `plpy.spiexceptions.*`), `plpy.Error`, `plpy.Fatal`.
- Квотирование: `plpy.quote_literal(s)`, `plpy.quote_ident(s)`, `plpy.quote_nullable(s)`.
- `SD` — локальный для функции словарь (кэш между вызовами **одной** функции в рамках бэкенда).
- `GD` — глобальный словарь, разделяемый **между всеми** plpython-функциями в backend-процессе.

Пример с подготовкой и кэшем:
```sql
CREATE FUNCTION user_name(uid integer)
RETURNS text LANGUAGE plpython3u AS $$
    if "plan" not in SD:
        SD["plan"] = plpy.prepare("SELECT name FROM users WHERE id=$1", ["int"])
    row = plpy.execute(SD["plan"], [uid])
    return row[0]["name"] if row else None
$$;
```

### Роль в RAG/AI-стеке
Ядро «вычислительного слоя» БД: позволяет из SQL напрямую вызывать Python-библиотеки (в т.ч. BAML, `httpx`, ML-пайплайны), векторизовать/постобрабатывать данные рядом с `pgvector`/`pg_turboquant`, оркестрировать шаги RAG (извлечение → реранк → LLM-вызов) одной транзакцией без отдельного backend-сервиса.

### Требования
- **`shared_preload_libraries`**: **нет**. Интерпретатор Python подгружается по требованию при первом вызове plpython3u-функции.
- **`CREATE EXTENSION`**: **да**, `CREATE EXTENSION plpython3u;` (альтернативно — `CREATE LANGUAGE plpython3u;`).
- **Схема по умолчанию**: отдельной схемы нет — это процедурный язык; функции можно создавать в любой схеме БД.
- **Зависимости**: системная `libpython3` той версии, с которой собран PostgreSQL. Только суперпользователь может создавать функции (trusted-варианта `plpython3` в PostgreSQL 17 не существует).
- **Безопасность**: так как язык untrusted, функция может читать файлы, открывать сокеты, запускать процессы. Использовать `SECURITY DEFINER` с осторожностью; не давать `CREATE` прав обычным ролям.

### Установка в нашем образе
Пакет из репозитория PGDG (apt), отдельный от сервера:
```dockerfile
RUN apt-get update && apt-get install -y postgresql-plpython3-17
```
В базовый образ `supabase/postgres:17.6.1.148` plpython3u **предустановленным не является** (уточнить по конкретному manifest'у supabase — обычно ставится отдельно). Также потребуются Python-пакеты AI-стека — ставятся через `pip install` в системный `python3` (или в venv, добавленный в `PYTHONPATH` бэкенда).

### Версия
Входит в состав PostgreSQL, версия совпадает с сервером (**17.x**). Пакет: `postgresql-plpython3-17`.

## BAML (BoundaryML)

### Назначение
BAML («Basically A Made-up Language», ранее расшифровывалось как Boundary Modeling Language) — DSL и рантайм для **типобезопасного описания LLM-вызовов и агентов**. В `.baml`-файлах декларативно описываются: функции (входные/выходные типы), промпты (Jinja-шаблоны), клиенты (провайдеры/модели), политики ретраев. Рантайм парсит ответ LLM в строго типизированные объекты, валидирует схему и при необходимости повторяет запрос.

### Ключевой API (Python)
Это **Python-библиотека**, а не SQL-объект. Типовой цикл:
1. Описать вызовы в файлах `*.baml` (напр. `baml_src/`).
2. Сгенерировать Python-клиент кодогенератором (`baml dev` / CLI) → появляется пакет `baml_client` с объектом `b`.
3. Вызывать из кода / из plpython3u:
```python
from baml_client import b
resume = b.ExtractResume(text)        # синхронно; тип результата описан в .baml
result = await b.ExtractResume(text)  # асинхронный вариант
```
Низкоуровневый модуль **`baml_py`** (экспорты подтверждены из `engine/language_client_python/python_src/baml_py/__init__.py`):
- `BamlRuntime` — ядро рантайма (можно инстанцировать вручную из строк `.baml`/конфига, без кодогенерации).
- `ClientRegistry` — реестр клиентов/провайдеров ( overriding модели, fallback).
- `BamlStream`, `BamlSyncStream` — стриминг ответа LLM.
- `BamlCtxManager` — менеджер контекста вызова.
- Мультимодальные типы: `Image`, `Audio`, `Pdf`, `Video`.
- Сгенерированный клиент (`baml_client.b`) предоставляет типизированные методы по каждой функции из `.baml`.

### Роль в RAG/AI-стеке
- **Структурированный вывод**: LLM-ответы (извлечение сущностей, классификация, summarization, шаги рассуждения) сразу приводятся к объявленной схеме — типобезопасно, без ручного `json.loads` и хрупких регулярок.
- **Валидация схемы + retry**: при невалидном/неполном ответе рантайм по политике повторяет запрос (контролируемая отказоустойчивость RAG-пайплайна).
- **Единый источник правды для промптов** (`.baml` в репозитории), версионирование и A/B клиентов/моделей через `ClientRegistry`.
- Вызов прямо из функции plpython3u: SQL → Python → LLM → структурированный результат в строке таблицы.

### Требования
- **Это библиотека, не расширение PG**: `CREATE EXTENSION` **не нужно**, `shared_preload_libraries` **не нужно**.
- Нужен рабочий Python-интерпретатор, доступный из plpython3u (тот же, куда ставится BAML).
- Доступ к сети (API провайдеров LLM) и учетные данные/ключи (через переменные окружения бэкенда — `plpy` видит `os.environ`).
- Требуется Python ≥ 3.10 (см. `requires_python` пакета `baml-py`).

### Установка в нашем образе
```dockerfile
RUN pip install baml-py
```
> ⚠️ **Уточнить выбор пакета.** На PyPI присутствуют два пакета:
> - `baml-py` — **канонический Python-рантайм** (импортируется как `baml_py`), текущая версия **0.223.0**, `requires_python >=3.10`. Именно его импортирует сгенерированный `baml_client`.
> - `baml` — отдельный пакет, версия **0.19.1**, `requires_python >=3.8,<4.0` (legacy/метапакет; назначение отличается — **уточнить** перед закреплением в `requirements.txt`).
>
> В задании фигурирует `pip install baml`; фактически для работы нужен **`baml-py`**. Рекомендуется закреплять `baml-py`.

### Версия
- Python-рантайм **`baml-py` 0.223.0** (PyPI).
- Спецификация языка BAML ( релиз-тэги вида `baml-language-*`) — отдельный трек версий (~0.15.x на дату проверки). Для образа важна именно версия `baml-py`.
- Точные имена полей `retry_policy` (число попыток, стратегия) и детали `ClientRegistry` — **уточнить** в актуальных docs.boundaryml.com (сайт рендерится JS, через curl не извлекается; проверить в IDE/сгенерированном `baml_client`).

## Замечания и подводные камни
- **Безопасность plpython3u**: только суперпользователь создаёт функции. Не давайте `CREATE` ролям приложений; для делегирования используйте `SECURITY DEFINER`-функции с жёсткой валидацией аргументов.
- **Потокобезопасность/бэкенды**: каждый PostgreSQL-бэкенд — отдельный интерпретатор Python; `GD`/`SD` не разделяются между соединениями.
- **Версия Python**: libpython в `postgresql-plpython3-17` должна совпадать с Python, куда ставятся `baml-py` и прочие пакеты, иначе `import` упадёт. Проверить `python3 --version` в образе.
- **Сетевые вызовы LLM из БД**: долгоживущие запросы к LLM удерживают backend-соединение; стоит ограничивать таймауты и не вызывать BAML в триггерах на hot-path без `statement_timeout`.
- **BAML-пакет**: предпочесть `baml-py`; убедиться, что сгенерированный `baml_client` совместим по версии с `baml-py`.
